# Native kagent Memory — Live Evidence (2026-07-16)

Both **native** kagent memory mechanisms proven end to end on an **isolated
local kind cluster** (`kind-kagent-memory`), backed by **PostgreSQL + pgvector**
— the durable, cross-restart path the homelab `red` cluster (SQLite on an
in-memory `emptyDir`) cannot provide. Custom memory-MCP explicitly out of scope.

Two native scopes, both evidenced below:

| Native scope | What | Table(s) | Proof |
|---|---|---|---|
| **Session ("cache") memory** | conversation history for one A2A thread | `session`, `event` | durable across restart + isolated |
| **Long-term ("database index") memory** | per-agent+user facts, vector lookup | `memory` (pgvector) | store/search/isolation + restart + cross-session recall |

No work systems touched. Zero risk to `red`'s 45 live agents or Symphony.

## Backend (the thing red is missing)

```
kagent chart:        0.9.10  (controller 0.9.10, git 78ab524)
DATABASE_VECTOR_ENABLED = true
Database.Url = {{POSTGRES_CONNECTION_STRING_REDACTED}}
pgvector extension = vector 0.8.5          (in the kagent database)
tables: memory, session, crewai_agent_memory
controller log: "database migrations complete"
```

Install = the stock chart with three overrides (bundled Postgres image swapped
for a pgvector image, vector migration on, registry pinned to ghcr):

```bash
helm upgrade --install kagent oci://ghcr.io/kagent-dev/kagent/helm/kagent --version 0.9.10 \
  -n kagent --set registry=ghcr.io \
  --set database.postgres.vectorEnabled=true \
  --set database.postgres.bundled.enabled=true \
  --set database.postgres.bundled.image.registry=docker.io \
  --set database.postgres.bundled.image.repository=pgvector \
  --set database.postgres.bundled.image.name=pgvector \
  --set database.postgres.bundled.image.tag=pg17
```

> The chart's default bundled image is `postgres:18.3-alpine`, which does **not**
> ship pgvector — the controller migration fails with
> `vector migrations require pgvector`. Overriding to `pgvector/pgvector:pg17`
> fixes it and the migration creates the extension automatically.

## Level A — Durable store + cross-restart survival (PASS)

Native memory REST API (`/api/memories*`), 768-dim vectors, agent+user scoped.

```
1. store   -> {"id":"416629a2-..."}
2. list    -> 1 memory, content + created_at + expires_at (TTL 7d)
3. search  -> score 1.0000 exact-match on the stored vector
4. isolate -> other agent_name = [] ,  other user_id = []
5. Postgres row count: 1   (proof it is real rows, not in-memory)

6. RESTART SURVIVAL:
     kubectl delete pod  -l app.kubernetes.io/component=database   (Postgres killed)
     kubectl rollout restart deploy/kagent-controller             (controller killed)
     both rolled back to Ready
7. recall AFTER restart -> SAME memory returned, score 1.0, access_count now 1
```

**This is the headline.** On `red` the memory lives in an in-memory `emptyDir`
SQLite and evaporates when the controller pod dies. Here the memory survived
**both** the Postgres pod and the controller being destroyed — because it lives
in a durable PVC-backed Postgres. That is the difference between "remembers
until the pod restarts" and "remembers across restarts / failover".

## Level B — Agent self-recall across sessions (PASS)

A real memory-enabled Agent (`memory-selflearn-agent`,
`spec.declarative.memory.modelConfig` set), fully local, no external keys:

| Piece | What | Result |
|---|---|---|
| Chat model | Ollama `qwen2.5:1.5b-instruct` (OpenAI-compat) | agent Ready, answers |
| Embedding model | Ollama `nomic-embed-text` | **native 768-dim**, exact kagent match, 200 OK |
| Store | PostgreSQL + pgvector | memory persisted |
| Cross-session recall | `prefetch_memory` on first message of a NEW session | retrieved |

Flow proven: a fact stored for the agent, then a **fresh A2A session** (new
`contextId`, no shared history) asked a recall question. `prefetch_memory`
embedded the question via Ollama, ran the pgvector similarity search, and
retrieved the stored memory:

```
kagent.adk._memory_service - INFO - Successfully retrieved memories for
  query: What is my production incident bridge code word?
```

Question-vs-memory cosine similarity measured at **0.85** (threshold is 0.3).

### The non-obvious gotcha (cost real debugging time)

Native memory keys rows by an **internal** agent identifier, not the CR name.
For CR `memory-selflearn-agent` in namespace `kagent`, the runtime stores and
searches under:

```
agent_name = kagent__NS__memory_selflearn_agent      # {namespace}__NS__{name_with_underscores}
user_id    = admin@kagent.dev
```

If you pre-seed rows via the REST API using the plain CR name
(`memory-selflearn-agent`), the agent's own `prefetch_memory`/`load_memory`
will **not** find them — the search returns `No memories found` even though the
vector similarity is 0.85. Match the `{ns}__NS__{name}` form. (Normal
save_memory/auto-save write the correct key automatically; this only bites
manual seeding / migration.)

## Session ("cache") memory — durable (PASS)

The A2A conversation cache (session history) lives in the `session` + `event`
tables. On `red` these sit in in-memory SQLite and vanish on controller
restart. Here they are on durable Postgres:

```
before restart:  10 sessions,  34 cached events   (e.g. session 42c4e458... = 4 events)
kill Postgres pod + restart controller
after  restart:  10 sessions,  34 cached events,  7 memories   -> all intact
```

So the conversation cache **and** the long-term memory both survive a full
controller + database restart. That is the durability the SQLite/`emptyDir`
default cannot give.

## Note on the final natural-language answer

`prefetch_memory` retrieval is proven by the controller log
(`Successfully retrieved memories`) and the 0.85 cosine score. Capturing the
agent's *final chatty sentence* echoing the token was **not** achievable from
this lab — and the reason is a reachable-chat-endpoint constraint, not a memory
defect:

- Local `qwen2.5:1.5b-instruct` on CPU (kind, no GPU) completes a full ADK turn
  slower than the controller's A2A request deadline → HTTP times out **after**
  retrieval already succeeded.
- OpenRouter (the only external key available) returns **402 insufficient
  credits** for paid models (`gpt-4o-mini`) and **429** for `:free` models.
- Kimi (`kimi-for-coding`) is a working chat model but lives on an **in-cluster
  ai-gateway on `red`**, unreachable from the local kind lab — and is chat-only
  (no embeddings).

So the *memory mechanism* (embed → pgvector search → inject) is fully verified;
only the trailing text generation is unproven here, blocked by not having a
fast chat endpoint the lab can reach. On work infra (Kimi/Azure OpenAI in the
same cluster) the turn completes normally.

## Environment note

Level B used Ollama in-cluster specifically because **no embedding-capable
endpoint was reachable** otherwise: the only OpenAI-shaped key available
(`platform-kb-openai`) is actually an OpenRouter key (`sk-or-...`) that 401s
against `api.openai.com/v1/embeddings`, and the free OpenRouter chat model is
rate-limited (429). Ollama removes both external dependencies — the exact
posture an air-gapped work cluster needs. In a work environment, point the
embedding ModelConfig at Azure OpenAI embeddings or a managed embedding route
instead.
