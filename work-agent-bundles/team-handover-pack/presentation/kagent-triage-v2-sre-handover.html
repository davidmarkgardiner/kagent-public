Problem

Current risk

What changes

Every SRE session produces evidence: app onboarded, workflow used, feedback captured, and an improvement routed.

What engineering gets

Operating loop

The six-step adoption cycle.

01

02

First contact

03

Game-day

04

Work order

Recommended ticket sequence.

Priority Ticket Why it matters Done when
P0 Prepare variables and access Every bundle depends on MCP names, namespaces, datasource UIDs, approval route, and GitLab scope. Shared variables filled privately and blockers owned.
P0 SRE first-contact app onboarding Turns the platform into a concrete workflow around one app SRE cares about. First-contact output reviewed by SRE.
P1 First controlled game-day Forces the real question: did SRE use it, trust it, and improve it? Exercise completed or blocked with evidence.
P1 Feedback and improvement loop Prevents adoption feedback becoming meeting notes. At least one feedback item routed and owned.
P2 Adoption reporting Shows whether SRE is actually using the system. Dashboard, report, or fleet summary exists.
Game-day model

Start small, make it repeatable.

Cadence

Weekly for four weeks, 60-90 minutes, one SRE owner or pair, one app, one lower-env scenario.

Guardrails

Lower-env only, no production chaos by default, read-only triage agents, HITL or GitOps for remediation.

Grafana metrics/logs/traces, GitLab ticket/MR, KB lookup, eval score, feedback item, adoption report.
