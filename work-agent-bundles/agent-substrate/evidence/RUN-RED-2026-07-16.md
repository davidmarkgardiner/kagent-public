# Agent Substrate — Live Evidence on the `red` Cluster (2026-07-16)

The **real-world** run: no kind bootstrap, no installer script, no per-actor patching.
Substrate installed onto the existing **`red`** Kubernetes cluster (already running kagent
0.9.10 + 45 live agents + Symphony), then a `SandboxAgent` driven **purely by
`kubectl apply` of a CRD**. This is the shape a work rollout takes.

## How it was installed (4 charts, declarative)

`red` already ran kagent but **could not run a SandboxAgent** — it had
`sandboxagents.kagent.dev` (the declaration) but no `ate.dev` CRDs and no `ate-system`
(the runtime). Added the two missing Agent Substrate charts, then flipped kagent into
substrate mode:

```bash
# 1. ate.dev CRDs (workerpools.ate.dev, actortemplates.ate.dev)
helm upgrade --install substrate-crds \
  oci://ghcr.io/kagent-dev/substrate/helm/substrate-crds --version 0.0.6 \
  --namespace ate-system --create-namespace --wait

# 2. ate-system runtime (ate-api-server, atelet, atenet, rustfs, 6x valkey)
helm upgrade --install substrate \
  oci://ghcr.io/kagent-dev/substrate/helm/substrate --version 0.0.6 \
  --namespace ate-system --wait --timeout 12m

# 3. flip the EXISTING kagent into substrate mode (reuse-values preserves all config)
helm upgrade kagent oci://ghcr.io/kagent-dev/kagent/helm/kagent --version 0.9.10 \
  --namespace kagent --reuse-values --wait \
  --set controller.substrate.enabled=true \
  --set controller.substrate.ateApiEndpoint=dns:///api.ate-system.svc:443 \
  --set controller.substrate.ateApiInsecure=true \
  --set substrateWorkerPool.create=true \
  --set substrateWorkerPool.replicas=1 \
  --set substrateWorkerPool.ateomImage=ghcr.io/kagent-dev/substrate/ateom-gvisor:v0.0.6
```

**No golang-adk patch needed on red** — red's kagent already sets `registry: ghcr.io`, so
the controller wrote a working `ghcr.io/kagent-dev/kagent/golang-adk@sha256:4683a34e...`
into the ActorTemplate. (This is the permanent, source-level version of the workaround the
kind run needed — proof that `registry=ghcr.io` fixes it declaratively.)

## Then, purely declarative — apply the CRD

```bash
kubectl apply -f examples/sandboxagent-hello.yaml   # SandboxAgent, no script, no patch
```

## API invocation finding (important for work integration)

Verified against the live controller on `red` (kagent 0.9.10):

| agent | `workloadMode` | `POST /api/a2a/kagent/<name>` (message/send) |
|---|---|---|
| `k8s-agent` (normal) | `deployment` | **200 — real LLM answer** returned |
| `hello-substrate` (substrate) | `sandbox` | **404 — "Agent not found"** |

- The substrate **runtime** is fully API-verified (Ready + actor Resume/Suspend RPCs + snapshot).
- But a **SandboxAgent is NOT invocable via the controller's A2A REST endpoint** that serves
  normal agents. Sessions can be created for it (`POST /api/sessions` with
  `agent_ref=kagent__NS__hello_substrate` → 200), but no controller message/task route serves
  it — sandbox actors are reached through the substrate data plane (atenet → the actor's own
  HTTP endpoint), which the UI/controller uses internally.
- **Implication:** if work needs to drive substrate agents **programmatically** (e.g. from the
  Alertmanager → Kafka → triage pipeline over HTTP/A2A), the standard A2A REST path does not
  reach them in 0.9.10. Deployment-mode agents remain the API-drivable option today; substrate
  agents currently assume UI/data-plane invocation. Track upstream for A2A exposure of
  `workloadMode: sandbox`.

## What this proves for work

- The install is the **same 4 charts** on a real, already-running kagent cluster — additive.
  The 45 existing agents kept running through the kagent helm upgrade (controller rolled,
  agents untouched).
- Once installed, a substrate agent is **pure `kubectl apply` of a `SandboxAgent` CRD** —
  GitOps-friendly, no imperative steps, no per-actor patching.
- gVisor checkpoint/restore + suspend/resume works end-to-end and the actor idles at ~0 RAM
  with its state snapshotted.

## Rollback

```bash
helm -n kagent rollback kagent 2                 # back to pre-substrate revision
kubectl delete sandboxagent hello-substrate -n kagent
helm -n ate-system uninstall substrate substrate-crds
kubectl delete ns ate-system
```

The pre-upgrade kagent values were captured to `red-kagent-values-backup.yaml` before the
change.
