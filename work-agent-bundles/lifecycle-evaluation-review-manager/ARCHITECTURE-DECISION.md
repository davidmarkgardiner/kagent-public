# Architecture Decision: Inline Versus Separate Evaluator

## Decision

Use a separate reusable evaluator step, implemented with an inline public-image
runtime.

```text
incident workflow
  -> verification
  -> ticket update
  -> templateRef: agent-lifecycle-eval
       runtime: alpine:3.19
       scripts: ConfigMap-mounted from GitOps
       permissions: evaluator-only
  -> ticket close only if eval passed
```

## Why Not Fully Inline In Every Workflow?

Fully inline means copying scorer logic into every incident workflow. That is
easy to start, but it creates drift:

- metric names can diverge
- hard gates can differ between workflows
- security fixes have to be copied everywhere
- review and promotion become harder

## Why Not A Custom Evaluator Container?

A custom image is clean from a software packaging perspective, but it creates
friction in environments where teams cannot easily build, push, or approve new
containers.

The current work-friendly compromise is:

- public image: `alpine:3.19`
- runtime packages: `python3`, `py3-yaml`
- scorer scripts mounted from `agent-eval-runtime-files` ConfigMap
- ConfigMap generated by `kubectl apply -k observability/agent-evals`

## Why A Separate Evaluator Step?

The evaluator should be independently governed from remediation:

| Concern | Remediation step | Evaluation step |
| --- | --- | --- |
| Can mutate cluster | Yes, after HITL and with narrow workflow permissions | No |
| Can update ticket | Yes, in ticket-update step | Prefer no; can output report for separate ticket step |
| Reads secrets | Only if needed by remediation/ticket update | No |
| Scores hard gates | No | Yes |
| Publishes metrics | Optional | Yes |

## Implementation Contract

Install:

```bash
kubectl apply -k observability/agent-evals
```

Call from workflow:

```yaml
templateRef:
  name: agent-lifecycle-eval
  template: evaluate-lifecycle
```

Pass:

```yaml
- name: workflow_json
  value: "{{tasks.prove-result.outputs.parameters.workflow-json}}"
- name: marker_evidence
  value: "{{tasks.prove-result.outputs.parameters.marker-evidence}}"
```

Output:

- lifecycle run JSON
- lifecycle summary Markdown
- Prometheus text metrics
- non-zero task exit if score fails or hard gate fails

## Acceptance Criteria

- The evaluator is reusable across incident workflows.
- The evaluator does not require a custom container image.
- The evaluator does not own remediation permissions.
- Ticket closure can be gated on evaluator pass/fail.
- The same `metrics.py` library is used by offline and online metric emission.
