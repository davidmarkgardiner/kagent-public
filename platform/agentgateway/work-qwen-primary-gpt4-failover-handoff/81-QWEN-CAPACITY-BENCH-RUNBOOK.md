# Qwen Capacity Bench Runbook

## Goal

Find the practical alert-to-agent ceiling by sending concurrent requests to the
kagent A2A API. This is the closest lightweight replica of real alert-triggered
workflows: alerts/events create kagent calls, kagent calls the model path, and
the task must reach A2A state `completed`.

The agentgateway benchmark remains useful as a lower-level diagnostic, but the
first capacity number should come from the kagent-facing API.

## Prerequisites

- Qwen token refresher has written a valid provider token.
- kagent `ModelConfig` or agentgateway route points to Qwen only for this test.
- Prometheus/Alloy is scraping agentgateway data-plane metrics.
- Loki or gateway logs are available for reset/timeout evidence.
- No automatic GPT-4 fallback is enabled during capacity discovery.
- The target kagent agent is Ready and points at the Qwen model config.

## First Test: 20 Concurrent Kagent Calls

Run this first. It sends 20 requests at once to the kagent A2A endpoint and
checks that the A2A task state is `completed`.

```bash
TOTAL=20 \
CONCURRENCY=20 \
AGENT_NAME=<agent-name> \
PROMPT="reply with exactly: ok" \
./bench-kagent-a2a.sh
```

If the work agent already has an ingress, tunnel, or port-forward, skip the
script-managed port-forward:

```bash
KAGENT_BASE_URL="https://{{KAGENT_HOSTNAME}}" \
TOTAL=20 \
CONCURRENCY=20 \
AGENT_NAME=<agent-name> \
./bench-kagent-a2a.sh
```

Success means:

```text
status_counts includes 200:20
state_counts includes completed:20
completed_rate = 1.0
```

The benchmark default `CURL_TIMEOUT` is 300 seconds. This is deliberate: kagent
can return a controller-side `500` just after its own long-running downstream A2A
timeout, and a shorter curl timeout can hide that as transport status `000`.

## Run The Sweep

After the first 20-concurrent test, run the kagent sweep:

```bash
CONCURRENCY_LEVELS="1 2 4 8 12 16" \
REQUESTS_PER_LEVEL=40 \
MIN_COMPLETED_RATE=0.99 \
MAX_P95_SECONDS=180 \
AGENT_NAME=<agent-name> \
./capacity-sweep-kagent-a2a.sh
```

If the work agent already has an ingress, tunnel, or port-forward, skip the
script-managed port-forward:

```bash
AGENTGATEWAY_BASE_URL="https://{{AGENTGATEWAY_HOSTNAME}}" \
CONCURRENCY_LEVELS="1 2 4 8 12 16" \
REQUESTS_PER_LEVEL=40 \
./capacity-sweep-agentgateway.sh
```

Use the gateway sweep only if you need to isolate whether failures are in
kagent/controller/A2A or in gateway/provider/model capacity.

If all levels pass, extend:

```bash
CONCURRENCY_LEVELS="24 32 48 64" \
REQUESTS_PER_LEVEL=60 \
MIN_COMPLETED_RATE=0.99 \
MAX_P95_SECONDS=180 \
AGENT_NAME=<agent-name> \
./capacity-sweep-kagent-a2a.sh
```

The script writes:

- `kagent-capacity-sweep.csv`
- one `summary.json` per concurrency level
- one raw response body per request
- one `summary.tsv` per concurrency level
- one `.state` file per request containing the A2A task state

## Single-Level Confirmation

After finding the first failing level, rerun the last passing level for longer:

```bash
TOTAL=200 CONCURRENCY=<last-passing-concurrency> AGENT_NAME=<agent-name> ./bench-kagent-a2a.sh
```

Then rerun the first failing level:

```bash
TOTAL=100 CONCURRENCY=<first-failing-concurrency> AGENT_NAME=<agent-name> ./bench-kagent-a2a.sh
```

Do not set production capacity from a single short run.

## What Counts As Failure

Treat any of these as capacity pressure:

- HTTP `429`
- HTTP `5xx`
- curl status `000`
- A2A state other than `completed`
- connection reset
- timeout/deadline/canceled reason
- p95 latency above the workflow deadline budget
- kagent/A2A call did not complete
- queue backlog keeps growing after the test stops

## Context-Size Follow-Up

Once 20 concurrent requests complete, increase prompt size before increasing
agent count. Use the same 20-concurrent run with progressively larger prompts:

```bash
PROMPT="$(python3 - <<'PY'
print('Summarise this alert context and reply with exactly: ok. ' + ('namespace=test pod=example condition=warning logs=sample ' * 200))
PY
)" TOTAL=20 CONCURRENCY=20 AGENT_NAME=<agent-name> ./bench-kagent-a2a.sh
```

Record the prompt size, p95 latency, and completed rate. This tells us whether
context size, not raw concurrency, is the limiting factor.

## PromQL During Test

Gateway request rate generated by kagent calls:

```promql
sum by (route, backend, status, reason) (
  rate(agentgateway_requests_total{route=~".*llm.*"}[1m])
)
```

429s:

```promql
sum by (route, backend, reason) (
  increase(agentgateway_requests_total{route=~".*llm.*",status="429"}[5m])
)
```

Timeouts/resets if exposed as reasons:

```promql
sum by (route, backend, reason) (
  increase(agentgateway_requests_total{route=~".*llm.*",reason=~"Timeout|DeadlineExceeded|Canceled|Connection.*|Reset.*"}[5m])
)
```

Latency:

```promql
histogram_quantile(
  0.95,
  sum by (route, backend, le) (
    rate(agentgateway_request_duration_seconds_bucket{route=~".*llm.*"}[5m])
  )
)
```

## How To Pick The Limit

Example:

```text
Concurrency 8 passes.
Concurrency 12 shows 429s and p95 > 90s.
safe_qwen_concurrency = 8
configured_limit = floor(8 * 0.7) = 5
```

Set Argo/Kafka controls to 5 active Qwen calls until more evidence exists.
