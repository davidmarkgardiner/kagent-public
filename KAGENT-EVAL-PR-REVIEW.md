# Kagent Eval PR Review Request

## Review Goal

Review the lifecycle evaluation implementation for correctness, portability,
and operational gaps before this pattern is lifted into a work environment.

Focus on bugs and risks, not style nits.

## Scope

Review these files:

```text
observability/agent-evals/
a2a/smart-triage-fanout-demo/workflow.yaml
a2a/smart-triage-fanout-demo/README.md
observability/README.md
DEMOS.md
README.md
chaos/litmus/scripts/chaos-eval-poc.py
KAGENT-EVAL-LIFT-AND-SHIFT-HANDOFF.md
```

Do not review unrelated dirty files unless they affect the eval path.

## Intended Behavior

The smart-triage demo should now prove this shape:

```text
normalize incident
  -> A2A fan-out to Kubernetes/network/Grafana/GitOps specialists
  -> incident commander synthesis
  -> HITL suspend/resume
  -> proof markers
  -> agent-lifecycle-eval WorkflowTemplate
  -> lifecycle score and hard-gate verdict
```

The runtime should not require a custom container image. It should use
`alpine:3.19`, install `python3` and `py3-yaml`, and mount the scorer scripts
from the `agent-eval-runtime-files` ConfigMap generated by Kustomize.

## Review Questions

1. Does the Argo wiring work without an artifact repository?
2. Does `kubectl kustomize observability/agent-evals` render a valid
   ConfigMap and `WorkflowTemplate`?
3. Does `a2a/smart-triage-fanout-demo/workflow.yaml` correctly pass marker and
   workflow evidence into `agent-lifecycle-eval`?
4. Are hard gates correct for real operations: HITL before remediation,
   verification before success, ticket update before close, right namespace?
5. Are synthetic public-demo markers clearly separated from real work evidence?
6. Is any public artifact leaking secrets, private IPs, tenant/subscription IDs,
   internal URLs, or real tokens?
7. Are dashboard and alert metrics named consistently with
   `summarize-agent-scores.py`?
8. Is the lift-and-shift handoff complete enough for another agent/team to
   implement this in a different environment?
