Observability event management

Use Vector to clean the bus, not replace the workflow.

The current Confluent proof already moves Kubernetes and alert events through Kafka into Argo. Vector is the missing middle layer for normalization, filtering, and short-window dedupe before Argo creates Workflows.

Use

Vector as an event normalizer between raw Confluent topics and triage topics.

Keep

Argo Sensors as the Kubernetes-native trigger boundary for Workflow creation.

Avoid

Treating Vector dedupe as durable 24-hour incident memory on its own.

Recommended shape

Raw topics stay auditable. Triage topics become automation-ready.

Vector consumes raw records, applies source-specific parsing and a common schema, then publishes smaller normalized records for Argo.

Dedupe stance

Suppress noise early, keep durable incident memory later.

Vector can prevent repeated events from flooding Argo, but it should not be the sole system of record for high-value incident grouping.

Use Vector dedupe for

  • Short-window suppression before Workflow objects are created.
  • Field-based dedupe on cluster, namespace, name, and reason.
  • Reducing repeat triage noise from known noisy Kubernetes events.

Keep durable dedupe for

  • 24-hour incident grouping and operator-facing history.
  • Human-in-the-loop remediation safety decisions.
  • Restart-proof dedupe where losing memory would create risk.
First implementation rule: Vector may reduce duplicate submissions, but Argo or the incident layer still needs the final "have we already acted on this?" guard.
Normalized event

Give Argo one small envelope, not every producer's native shape.

This is the shape Argo Sensors should filter and Workflows should receive. The raw payload remains attached for debugging. 

schema_version: observability.triage.v1
source: alloy | alertmanager | grafana
cluster: {{CLUSTER_NAME}}
environment: {{ENVIRONMENT}}
severity: critical
event_type: kubernetes-event | prometheus-alert | grafana-alert
reason: CrashLoopBackOff
object_kind: Pod
namespace: default
name: example-pod
dedupe_key: {{CLUSTER_NAME}}:default:Pod:example:CrashLoopBackOff
raw_topic: k8s-events
raw: { original source payload }
Spike path

Start small with alerts, then extend to Kubernetes events.

Alertmanager records have a smaller shape than OTLP Kubernetes events, so they are the safer first proof.

1

Deploy Vector

Management cluster, read raw topic, write triage topic.

2

Normalize alerts

Map Alertmanager records into the triage envelope.

3

Add dedupe key

Derive a stable key from source, alert name, cluster, and labels.

4

Publish triage

Write to alertmanager-events-triage.

5

Point Argo

Use the normalized topic in a new Sensor path.

6

Measure

Compare workflow count, duplicates, and readability.

Operator takeaway

Vector improves Confluent as an event bus.

The first valuable milestone is a cleaner, normalized, less noisy event stream feeding the same Argo workflow boundary.
