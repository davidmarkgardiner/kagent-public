Stakeholder view

Alert Manager to agentic triage

The proven path sends alerts from Alertmanager or Grafana directly into a Vector HTTP webhook receiver. Vector becomes the cleanup boundary: it filters noise, suppresses duplicates, enriches the event, and publishes only actionable records to Kafka for Argo to trigger agent workflows.

Core message: Kafka is the durable event bus, Vector protects the queue from noise, Argo starts controlled workflows, and specialist agents produce evidence in GitLab or ServiceNow.

Alertmanager/Grafana -> Vector webhook receiver -> Vector cleanup -> Kafka topic(s) -> Argo Sensor -> Agents -> GitLab / ServiceNow

What we are proving

Resolved, duplicate, malformed, and ignored alerts are dropped, counted, or quarantined by Vector.

Durable routing Kafka carries the cleaned contract

One topic can work initially; multiple topics can split app, platform, security, or compliance routes later.

GitLab or ServiceNow records capture triage, remediation, approvals, and evidence.

High-level design

The current target route posts alerts directly into Vector's webhook receiver. Vector then publishes the cleaned actionable event directly to Kafka, where Argo watches the topic or topics.

Ownership by layer

Vector owns cleanup

  • Normalizes Alertmanager and Grafana payload shapes into one triage envelope.
  • Drops resolved alerts, ignored severities, malformed payloads, and known-noise labels.
  • Builds stable route fields: target agent, route key, dedupe key, and automation policy.
  • Suppresses duplicates before the actionable topic and before Argo workflow creation.

Kafka owns durable delivery

  • Starts with one actionable topic if that is simplest.
  • Can split later into app, platform, security, compliance, or priority topics.
  • Allows replay and audit without asking Argo to process raw noise.
  • Gives Argo Sensors a clean stream to watch.

Argo and agents own action

  • Argo Sensors watch the cleaned topic or topics and create workflow runs.
  • Workflows collect evidence, call agents, and enforce approval gates.
  • Agents can hand off to specialists based on route and skill fit.
  • Final records are written to GitLab and optionally ServiceNow.

Ticket and approval loop

GitLab evidence

Creates an issue or merge request with alert fingerprint, affected workload, agent route, triage summary, logs, metrics, and recommendation.

Human approval

Required before write-capable actions such as command execution, policy changes, or merge request approval.

ServiceNow escalation

Used when a formal incident is required, ownership is unclear, the risk is high, or the workflow cannot safely remediate.

GitLab ticket should show: alert fingerprint -> affected service -> selected agent -> evidence -> triage -> remediation recommendation/action -> approval state -> final outcome
