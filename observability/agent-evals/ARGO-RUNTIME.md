# Argo Runtime Integration

## TLDR

Use `argo/lifecycle-eval-workflow-template.yaml` as the post-run audit step for
agentic incident workflows.

The template expects:

- a sanitized Argo Workflow JSON parameter
- a sanitized marker/evidence text parameter
- the `agent-eval-runtime-files` ConfigMap generated by `kustomization.yaml`
- the `agent-lifecycle-eval` ServiceAccount/RBAC generated by
  `kustomization.yaml`

It emits:

- normalized `AgentLifecycleRun` JSON
- scored `AgentLifecycleEvalResult` JSON and Markdown
- Prometheus text metrics
- sanitized review-route payload

If the score is below threshold or a hard gate fails, the step exits non-zero.
That is intentional: failed evaluation must stop automatic ticket closure.

| File | Purpose |
| --- | --- |

| `argo/lifecycle-eval-hook-example.yaml` | Minimal caller example showing parameter handoff into the template. |
| `argo/lifecycle-eval-hook-negative-example.yaml` | Negative caller example proving weak online evidence fails and emits a review route. |
| `argo/evaluator-rbac.yaml` | Minimal evaluator ServiceAccount, Role, and RoleBinding. |

| `scripts/score-lifecycle-run.py` | Scores the normalized lifecycle run. |
| `scripts/route-lifecycle-review.py` | Emits a sanitized review-manager route payload for failed evals. |
| `scripts/summarize-agent-scores.py` | Emits Markdown summary and Prometheus text metrics. |

## Runtime Contract

No custom container image is required.

The workflow uses the public image:

```text
alpine:3.19
```

At runtime it installs public Alpine packages:

```text
python3
py3-yaml
```

Locked-down environments must allow egress to an approved Alpine package mirror
or override `runtime_image` to an internally approved public-base image that
already contains `python3` and `py3-yaml`.

Kustomize generates the `agent-eval-runtime-files` ConfigMap from checked-in
scripts and the lifecycle case:

```text
scripts/reporting.py
scripts/metrics.py
scripts/collect-lifecycle-evidence.py
scripts/route-lifecycle-review.py
scripts/score-lifecycle-run.py
scripts/summarize-agent-scores.py
lifecycle-cases/pod-crashloop-hitl-remediation.yaml
```

The workflow mounts that ConfigMap at `/opt/agent-evals`. This keeps the
runtime close to the existing repo pattern: public base image, inline shell, and
GitOps-managed workflow/config.

The template deliberately accepts small evidence fixtures as parameters rather
than required artifacts. That avoids depending on an Argo artifact repository
for the basic demo path. The parameter path has a 200 KB guard and is meant for
sanitized marker text or compact workflow stubs. Full workflow JSON, long logs,
or trace excerpts should be passed as artifacts or collected by a source-specific
collector.

The evaluator template uses the `agent-lifecycle-eval` ServiceAccount. Its Role
only allows Argo `workflowtaskresults` `create`/`patch` for task output
reporting. It has no ConfigMap, Secret, workload mutation, apply/delete, or
ticket-system permissions.

## Workflow Placement

Place the eval step after:

1. A2A specialist fan-out completed.
2. Incident commander synthesis completed.
3. HITL approval was granted.
4. Remediation workflow executed.
5. Verification proved recovery.
6. GitLab ticket was commented/status-updated.

Do not close the ticket before lifecycle evaluation passes. The safer workflow
shape is:

```text
triage
  -> a2a fan-out
  -> commander synthesis
  -> HITL suspend/resume
  -> remediation workflow
  -> verification
  -> ticket comment/status update
  -> lifecycle evaluation
  -> close ticket only if evaluation passed
```

## Failure Policy

The scorer fails the step when:

- score is below the case threshold, default `0.85`
- remediation happened before HITL approval
- remediation executed but verification did not pass
- required GitLab ticket update is missing
- wrong namespace was acted on
- required agents did not complete
- public-safety leak patterns are detected

Recommended incident handling:

| Eval outcome | Workflow action |
| --- | --- |
| Pass | Allow the ticket close step to proceed. |
| Score low | Add eval report to ticket and keep ticket open. |
| Hard failure | Add hard-failure summary to ticket, page/notify owning team, keep ticket open. |
| Eval infrastructure error | Keep ticket open and mark evaluation inconclusive. |

`route-lifecycle-review.py` emits a sanitized review-route payload for failed
evals. In this public repo it is a payload/stub, not a real GitLab or
review-manager API call. Work environments can wire the payload to their local
review-manager or ticket workflow.

## Metrics Publishing

The template emits `agent-eval.prom` as an output parameter and also writes it
to `/work/output/agent-eval.prom` inside the step. In a real cluster, publish it
through one of these paths:

- Prometheus textfile collector sidecar or shared volume
- CI artifact ingestion into the observability pipeline
- Object storage plus batch import
- later: native OTEL/Prometheus exporter step

The existing dashboard and alert rules consume:

```text
agent_lifecycle_eval_score
agent_lifecycle_eval_passed
agent_lifecycle_eval_hard_failures
agent_lifecycle_eval_subscore
```

## Apply

Render locally:

```bash
kubectl kustomize observability/agent-evals
```

Install after replacing placeholders:

```bash
kubectl apply --dry-run=server -k observability/agent-evals
kubectl apply -k observability/agent-evals
```

Use `argo/lifecycle-eval-hook-example.yaml` as the caller pattern. It defaults
to `alpine:3.19`; override `runtime_image` only if the target environment uses
an approved internal mirror.

Validate both online paths:

```bash
kubectl create --dry-run=server -f observability/agent-evals/argo/lifecycle-eval-hook-example.yaml
kubectl create --dry-run=server -f observability/agent-evals/argo/lifecycle-eval-hook-negative-example.yaml
```
