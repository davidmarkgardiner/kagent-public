Kagent evaluation design

Separate offline confidence from online control.

Offline eval is the replay and promotion path. Online eval is the runtime audit and gating path. Non-prod and prod are environments; offline and online are evaluation modes.

Offline = CI, replay, golden cases Online = Argo post-run audit gate Prod starts audit-only Metrics library is independent

MS-facing summary

We use offline evaluation before promotion and online evaluation after real workflow execution. Both can run in non-prod. Production online eval should start as audit-only, then gate write-capable remediation and ticket closure once evidence is stable.

Two evaluation modes

The same scoring contract is used in both modes. The difference is whether the run is captured/replayed or part of the live incident path.

Offline evaluation

Replay and promotion gate

Scores known cases without affecting a live workflow. Best for CI, prompt regression, model comparison, and sanitized production replay.

A
Golden caseExpected diagnosis, evidence, tools, and hard gates.
B
Captured runAgent response, tool calls, or normalized lifecycle JSON.
C
ScoreJSON, Markdown, Prometheus metrics, CI verdict.
Online evaluation

Runtime audit and gate

Scores the actual incident workflow after A2A, HITL, remediation, verification, and ticket update. Can audit first, then gate.

A
Argo workflowSmart triage emits sanitized markers and workflow JSON.
B
Lifecycle evalPublic Alpine runtime mounts scripts from ConfigMap.
C
ControlLow score blocks auto-close on high-risk paths.

End-to-end runtime flow

The evaluator should sit after verification and ticket update. It scores whether the workflow can safely claim success.

Rollout model

Start broad in non-prod, then introduce production gating only where the workflow can change systems or close tickets.

Stage 0

Local design

  • Golden casesDefine the expected diagnosis, evidence, tools, lifecycle steps, and threshold before any live workflow is involved.
  • Sample lifecycle runsUse public-safe JSON to prove the scorer can pass a good incident and explain the score.
  • Hard-gate testsProve bad behavior fails: wrong namespace, remediation before HITL, missing verification, or missing ticket update.
Stage 1

CI gate

  • Offline onlyRun against captured or synthetic data so CI never touches a real cluster or incident ticket.
  • Prompt/tool regressionFail changes that remove required evidence, call forbidden tools, or drift from the expected diagnosis.
  • Promotion verdictUse the score as a release check before an agent, prompt, or workflow is promoted.
Stage 2

Non-prod online

  • Every simulationScore each dev/staging incident workflow to prove the runtime handoff works end to end.
  • Chaos runsUse controlled failures to test whether agents gather evidence, request approval, and verify recovery.
  • Workflow proofConfirm the Argo evaluator runs after verification and emits JSON, Markdown, and metrics.
Stage 3

Prod audit

  • Audit-only firstScore real production runs without blocking the workflow while thresholds and evidence quality settle.
  • DashboardsTrack score trends, pass rate, hard failures, and agent drift across the fleet.
  • Low-score reviewInvestigate misses, add new golden cases, and improve prompts or workflow steps.
Stage 4

Prod gate

  • Write-capable flowsGate remediation and any workflow that can apply, merge, execute, or alter platform state.
  • Ticket closureOnly close the incident when remediation, verification, and ticket hygiene all pass evaluation.
  • Hard failures blockKeep the ticket open and attach the failure summary when safety or verification gates fail.

Metrics and access control

Metric generation is an independent library so offline reports, online Argo steps, and future exporters share the same naming and label rules.

scripts/metrics.py Metric contract

Owns metric names, label escaping, booleans, and Prometheus text rendering.

agent_lifecycle_eval_score Runtime score

Normalized from 0.0 to 1.0. Dashboards may display it as 0 to 10.

agent_lifecycle_eval_hard_failures Gate signal

Any non-zero value is a strong audit signal for write-capable flows.

Area Allowed Forbidden
Metric labels agent, case_id, run_id, workflow_name, dimension prompt text, ticket body, raw logs, private IPs, URLs, tokens, tenant IDs
Evaluator RBAC Read provided evidence, read ConfigMap scripts, write own outputs Cluster mutation, secret reads, GitLab write, broad pod logs
Dashboard access Aggregated scores, hard-failure counts, fleet status Raw workflow artifacts unless explicitly authorized

The evaluator scores the result. It must not own remediation or ticket mutation permissions. Keep those as separate workflow service accounts.
