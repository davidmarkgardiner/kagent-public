Kagent knowledge base POC

Git-backed docs, served as a queryable kagent tool.

A proof of concept that turns platform documentation into a doc2vec vector database, exposes it through querydoc MCP, and lets a kagent agent answer with cited source paths.

doc2vec querydoc MCP RemoteMCPServer platform-knowledge-agent

What the README explains

The POC is built from a small set of reviewable parts.

Seed corpus

docs/platform-kb contains the source docs and index used by the demo.

Indexer

build-platform-kb-db.sh runs doc2vec and refuses to publish a DB if embeddings fail.

Kubernetes runtime

k8s/ defines the PVC, querydoc service, RemoteMCPServer, ModelConfig, and agent.

Deployment script

deploy-poc-demo.sh applies resources, seeds the PVC, restarts querydoc, and verifies rollout.

Evidence capture

Evidence scripts record dry-run checks, live health, agent card, and A2A proof markers.

Credential isolation

The demo uses dedicated platform-kb-openai secret and ModelConfig, separate from shared defaults.

Why this pattern

Pre-indexing is simpler than asking every chat session to clone and search Git.

Direct Git pull inside the chat agent

Every session needs Git credentials, network access, parsing/indexing logic, and working-tree lifecycle handling. It also makes freshness and repeatability session-specific.

doc2vec + querydoc MCP

Indexing happens once on a schedule or manual trigger. The chat agent stays read-only and calls one tool: query_documentation.

The chat agent answers from a shared, refreshed search surface instead of becoming a repo crawler.

Runtime flow

From Markdown to cited answer in five steps.

01

Docs live in Git

Markdown remains the authoring surface for platform guidance.

02

doc2vec embeds

The indexer chunks docs and writes vectors into SQLite.

03

PVC stores DB

platform-kb.db persists independently of querydoc pods.

04

querydoc serves MCP

The MCP server exposes query_documentation.

05

Agent answers

kagent calls the tool and cites retrieved source paths.

docs/platform-kb -> doc2vec -> platform-kb.db -> querydoc MCP -> RemoteMCPServer -> platform-knowledge-agent

Safety model

The demo is intentionally non-provisioning.

It creates Kubernetes resources only in the kagent namespace. It does not create Azure, ASO, or KRO cluster resources.

Control Implementation
Azure/KRO guard Validation scans rendered YAML for provisioning resource kinds.
CronJob safety Nightly indexer is rendered with suspend: true.
Secret isolation POC uses dedicated platform-kb-openai secret.
DB integrity Build fails if doc2vec logs embedding errors.

Failure modes

Known failure modes are explicit and observable.

Failure Expected behavior Review signal
Embedding key missing or invalid Indexer/querydoc fail fast; existing DB remains untouched. Build logs contain embedding error; DB is not published.
Docs repo unavailable Indexer job fails; querydoc keeps serving last successful DB. CronJob failure with unchanged PVC content.
No matching docs Agent says the docs do not answer and routes to support process. Final response avoids unsupported remediation.
Stale docs reported Agent summarizes the gap; chat agent does not push Git. Separate approved stale-doc workflow owns PR creation.
