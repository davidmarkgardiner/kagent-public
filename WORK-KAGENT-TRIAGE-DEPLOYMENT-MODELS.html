Kagent triage: worker-local vs management-cluster routing

The practical security question is not where the agent pod runs. It is which credentials, tool grants, network paths, approval gates, and audit controls it is allowed to use. If the management-cluster agent uses AKS-MCP with admin rights back into the worker, it can have the same effective write blast radius as a local worker agent.

Short verdict

Path A can be the smaller exposed surface for local platform namespaces: events stay inside the worker cluster, the agent does local triage, and only controlled outbound traffic goes to the management agentgateway and GitLab.

Path B is useful for fleet-level triage, but its safety depends on tightly scoped central AKS-MCP credentials. If central AKS-MCP has broad admin on every worker, a management-plane compromise can fan out across the fleet.

Blast radius is credential-bound Central MCP needs hard scoping agentgateway stays on management

Architecture view: same management-hosted gateway and MCPs, different triage loop

Both models keep agentgateway and MCP services on the management cluster. The difference is whether the incident calls a local worker agent first, or leaves the worker through Kafka/Event Hub so a management-cluster agent can triage and then connect back to the worker through AKS-MCP.

What is exposed in both

  • Management agentgateway for LLM egress, policy, token metrics, and audit.
  • Management-hosted MCP services and tool servers, reachable only from authorized agents/workflows.
  • Outbound HTTPS 443 to GitLab for tickets or merge requests.

What changes

  • Path A calls the worker-local agent directly from local events.
  • Path B sends worker events out to Kafka/Event Hub first, then the management agent connects back to the worker through AKS-MCP.
  • Path B therefore has more cross-cluster edges to authenticate, monitor, and debug.

Blast Radius Comparison

If Path B grants central AKS-MCP broad admin rights on the worker cluster, then a compromised management agent can make the same destructive Kubernetes changes as a compromised worker-local agent. The blast radius becomes the granted RBAC scope, not the pod location.

Pros And Cons

Path A: worker-local kagent

Pros

  • Smallest dependency chain for worker-internal incidents.
  • No need to expose a public or shared agent endpoint for local event ingestion.
  • Events, logs, namespace state, and local tools are close to the failure domain.
  • Works better for critical local namespaces if management routing is down.
  • Can be made read-only by default, with write actions submitted as GitLab MRs or Argo workflows.

Cons

  • More agent deployments to operate across the fleet.
  • Needs consistent policy, prompt, ModelConfig, and audit configuration per worker.
  • Local service accounts must be carefully scoped so the agent cannot mutate broad cluster state.
  • Central dashboards need local audit and metrics export to compare agents consistently.

Path B: management-cluster kagent via Kafka/Event Hub

Pros

  • One central triage surface for many clusters.
  • Natural place for fleet correlation, shared Argo workflows, and centralized audit.
  • Good fit for application namespaces where events already flow through Confluent Kafka or Azure Event Hub.
  • Simpler to update one agent stack when the fleet does not need local fallback.

Cons

  • Remote AKS-MCP credentials can create the same worker-cluster blast radius as a local agent.
  • If one central identity can administer many workers, the aggregate blast radius may be larger.
  • More moving parts: producer, broker, EventSource, Sensor, workflow, MCP, gateway, and GitLab.
  • Higher latency, more network traffic, and often more tokens because the agent has to fetch remote context.
  • Management-cluster outages can break the triage path for the worker cluster.

Controls That Calm The Security Concern

1. Default triage agents to read-only

Give triage agents evidence tools only: get resources, logs, describe, YAML, events, connectivity, and configuration. Do not attach apply, patch, delete, exec, label, or annotate tools to normal triage agents.

2. Split agent front door from execution

Agents submit GitLab issues, GitLab MRs, or Argo workflows. Workflow service accounts hold mutation permissions and can require human approval before changes land.

3. Scope AKS-MCP per failure domain

For worker-local mode, scope MCP or tool server access to that worker and preferably to namespaces. For central mode, avoid one all-powerful identity across the fleet.

4. Lock network paths

Worker-local agents need outbound access to management agentgateway and GitLab, not broad inbound exposure. MCP endpoints should only accept traffic from authorized agent pods.

5. Keep agentgateway centralized

Route LLM traffic through the management agentgateway for model routing, cost tracking, token monitoring, policy, failover, and suspicious-use alerting.

6. Audit every hop

Carry a trace ID from event to agent run to MCP call to workflow to GitLab ticket. Alert on unexpected write tools, unusual token bursts, and cross-cluster MCP access.

Recommended Position

Use Path A first

For worker-local platform namespaces such as cert-manager, external-secrets, kro, kyverno, reloader, monitoring, and ingress-nginx.

Use Path B where it fits

For fleet-level correlation, application namespaces, and central operational queues already backed by Confluent Kafka or Azure Event Hub.

Use hybrid for critical services

Keep local fallback triage where losing management connectivity would block diagnosis of the worker cluster itself.

Do not grant blanket admin

The biggest risk is broad write-capable MCP access. That risk exists in both models and is worse when centralized across many workers.

Copyable Boss Briefing

The decision is not "agent in the worker cluster equals unsafe" and "agent in the management cluster equals safe." The real blast radius is the Kubernetes and MCP permission set the agent can use.

Path A has kagent running inside the worker cluster. The event stays local, triage runs local, LLM calls still go out through the management agentgateway for monitoring and token control, and the output goes to GitLab. This can be the smaller exposed surface because we do not need to expose an agent endpoint or route every local event through Kafka/Event Hub before diagnosis.

Path B has kagent running in the management cluster. Worker events go through Confluent Kafka or Azure Event Hub, then the management agent uses AKS-MCP to connect back to the worker cluster. If that AKS-MCP identity has admin rights on the worker, the management agent can do the same damage as a local write-capable agent. If the same identity reaches many workers, the aggregate blast radius can be larger.

The safer rollout is a hybrid: worker-local read-only triage for critical worker namespaces, central management triage for fleet-level and application events, and all write actions routed through GitLab MR or Argo Workflow with human approval. Keep agentgateway on the management cluster for LLM policy, token metrics, model routing, and audit. Lock down MCP endpoints so only authorized agents can call them, and avoid broad admin credentials in both models.
