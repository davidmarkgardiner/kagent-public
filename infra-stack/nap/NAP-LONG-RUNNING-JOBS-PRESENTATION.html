AKS NAP briefing

Preventing NAP evictions for long-running jobs

How to stop voluntary Karpenter consolidation from interrupting pods that need to run to completion.

AKS Node Auto-Provisioning Karpenter NodePool disruption Updated 2026-05-13
Problem

The node looks cheap to remove. The pod is still working.

What the team sees

A long-running Job pod is evicted while it is halfway through useful work.

What NAP sees

The node is empty enough or underutilized enough to be a consolidation candidate.

What is missing

The workload has not told Karpenter that this pod must run to completion.

Main answer

There is no global "only after jobs finish" switch.

Use workload and NodePool policy: protect the pod with karpenter.sh/do-not-disrupt, and run it on a batch pool that consolidates only when empty.
This controls voluntary NAP disruption. It is not a guarantee against expiration, node repair, VM failure, Azure maintenance, Spot eviction, or manual deletion.
Karpenter flow

Where disruption enters the lifecycle

With WhenEmptyOrUnderutilized, a running pod can still be on a node that Karpenter wants to consolidate.

Pod protection

Put the annotation on the pod template.

The Job object metadata is not enough. The running pod must carry the annotation.

apiVersion: batch/v1
kind: Job
metadata:
  name: long-running-task
spec:
  backoffLimit: 0
  template:
    metadata:
      annotations:
        karpenter.sh/do-not-disrupt: "true"
    spec:
      restartPolicy: Never
      containers:
        - name: worker
          image: example.azurecr.io/worker:latest
NodePool policy

Use a batch pool that waits for empty nodes.

WhenEmpty is the important change. It means consolidation waits until workload pods have left the node.

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: batch-jobs
spec:
  disruption:
    consolidationPolicy: WhenEmpty
    consolidateAfter: 10m
    expireAfter: Never
    budgets:
      - nodes: "1"
  template:
    metadata:
      labels:
        workload-type: batch
Policy matrix

Match disruption policy to workload behavior.

Workload Policy Why
stateless replicated services WhenEmptyOrUnderutilized, PDBs, multiple replicas Restart and reschedule are expected paths.
batch long-running jobs WhenEmpty plus do-not-disrupt Jobs should finish before voluntary movement.
ml expensive training Dedicated on-demand pool, no short expiry Restart cost is high and checkpoints may be sparse.
spot interruptible jobs Spot only with checkpointing and retries Spot can be interrupted outside NAP's voluntary controls.
Limits

What this protects and what it does not.

Protected

  • Voluntary consolidation
  • Routine optimization decisions
  • Drift in the normal case, unless NodeClaim termination grace changes the decision
  • Underutilized-node cleanup while the pod is active

Not protected

  • Expiration and node repair
  • VM failure or node not-ready events
  • Azure maintenance and platform interruption
  • Spot eviction
  • Manual node or NodePool deletion
  • Force deletion after NodePool termination grace
Karpenter cross-check

Upstream docs add two hard edges.

Termination grace

Once a node is already draining, NodePool terminationGracePeriod is a hard deadline. It can bypass PDBs and do-not-disrupt.

Forceful methods

do-not-disrupt does not exclude nodes from expiration, interruption, node repair, or manual deletion.

The duration form is valid in current upstream Karpenter, for example karpenter.sh/do-not-disrupt: "30m".
Work test

Verify in phases, starting read-only.

1. Baseline

Read current context, NodePools, nodes, and protected pods. Do not apply anything yet.

2. Dry-run

Validate the batch NodePool and Job manifests with server-side dry-run.

3. Controlled apply

After approval, run a small job in a test namespace and observe NAP events.

Running workloads on a NAP-enabled cluster can create or remove Azure VM capacity. Get approval for the test cluster, time window, quota impact, and cleanup owner.
Commands

Read-only baseline commands.

Use these first to understand whether the current policy is safe for long jobs.

kubectl config current-context
kubectl get nodepools -o wide
kubectl get nodepool default -o yaml
kubectl get nodes \
  -L karpenter.sh/nodepool,workload-type

kubectl get pods --all-namespaces \
  -o custom-columns='NAMESPACE:.metadata.namespace,NAME:.metadata.name,NODE:.spec.nodeName,DO_NOT_DISRUPT:.metadata.annotations.karpenter\.sh/do-not-disrupt' \
  | grep -v '<none>' || true
Evidence

Prove NAP skipped voluntary eviction.

Capture pod annotation, events, Karpenter logs, and node state before and after the job completes.

kubectl get events --all-namespaces \
  --sort-by='.lastTimestamp' \
  | grep -Ei 'karpenter|disrupt|consolidat|drift|nodeclaim' \
  | tail -50

kubectl logs -n kube-system \
  -l app.kubernetes.io/name=karpenter \
  -c controller --tail=200

kubectl wait -n nap-disruption-test \
  --for=condition=complete \
  job/long-running-task --timeout=70m
Decision

Recommended work standard

Long-running batch gets its own scheduling class: dedicated NodePool, WhenEmpty, do-not-disrupt, on-demand capacity, and evidence captured before rollout.
