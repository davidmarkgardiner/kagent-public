Work delivery · Evidence-first worker triage

Turn raw worker signals into safe, actionable human triage.

This work delivers a repeatable path for Kubernetes logs and events to arrive at a central, read-only triage agent with enough context to produce one clear GitLab ticket a human can act on—without making the path depend on Alertmanager or Grafana.

Delivery goal: every worker can send a redacted, bounded, correctly labelled evidence package to the management cluster. The management side correlates it, prevents duplicate ticket storms, asks a read-only agent to diagnose it, and presents the result in a human-readable GitLab ticket with the exact cluster, namespace, pod, container when known, signal and safe evidence.

Key placement decision: raw/noisy evidence is prepared locally in each worker before it crosses the cluster boundary. The agent, ticket credentials, durable incident state and policy stay central.

What the delivery should achieve

For SREs

Faster first understanding

A ticket says where the failure happened and what was observed, so the human starts with evidence rather than reconstructing it from dashboards and alerts.

For workers

Local safety before egress

Vector redacts, caps, normalises and buffers evidence locally. A worker outage or burst should not become uncontrolled agent work or a raw-log export.

For the platform

One governed triage plane

Management owns replay handling, durable duplicate control, read-only agent access, workflow audit and ticket creation—rather than reproducing those controls in every worker.

How the work agent delivers it

Reproduce safely

Deploy one additive, non-production worker path. Merge the Alloy fragment into the existing Alloy release; do not create a competing collector.

Prove evidence

Generate an approved application-log signal and Kubernetes event. Capture the payload at each boundary and verify field completeness and redaction.

Prove the human outcome

Confirm the central agent creates or updates one readable GitLab ticket, with safe evidence and practical reach-back details.

Scale deliberately

Add workers and namespaces only after replay, duplicate handling, access boundaries and ticket quality meet the desired-state gates.

What this explicitly does not do

Boundary

No autonomous remediation

The agent diagnoses and advises. It does not mutate worker clusters.

Boundary

No replacement for human paging

Alertmanager can continue to page humans. Grafana/Loki remain useful for dashboards and follow-up investigation.

Boundary

No blind fleet rollout

A Ready pod or rendered YAML is not success. Each gate needs direct evidence before another worker or namespace is onboarded.

Plain-English TL;DR

Instead of waiting for an alert to lose its useful context and then asking an agent to hunt it down, the worker sends a small, safe evidence package at the moment the log or event occurs. The central platform turns that into one useful ticket for a human—while preserving existing alerting and keeping the agent read-only.
