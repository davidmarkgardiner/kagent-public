MCP Tool Platform Pattern

Put the tools behind Agent Gateway

Kagent still defines agents and their intended tools. Agent Gateway becomes the runtime control point that authenticates the caller, filters tool discovery, and blocks calls outside the grant.

One-line explanation

ToolCatalogEntry and ToolGrant describe who should get what; Agent Gateway enforces it; kagent consumes only the gateway-fronted MCP endpoint.

The Moving Parts

The design splits source-of-truth objects from runtime enforcement. That keeps onboarding, audit, and policy review declarative, while avoiding trust in the agent process as the security boundary.

1

Tool is catalogued

A BYO MCP server is verified and written into ToolCatalogEntry.

source of truth
2

ToolGrant says exactly which agent can use which verified tools.

3

Gateway policy is rendered

A controller or GitOps renderer turns the grant into MCP auth policy.

projection
4

Kagent points at gateway

The agent uses a RemoteMCPServer URL for Agent Gateway, not the tool service.

controlled path
5

Discovery and calls are enforced

tools/list is filtered. tools/call is blocked if not granted.

runtime guard

Runtime Request Flow

The important change is the middle hop. Agents no longer talk directly to every MCP server. They talk to Agent Gateway, which fans out to the right MCP target only after policy is applied.

What Each Layer Owns

This is the easiest way to explain the split to reviewers: kagent remains the agent platform, ToolGrant remains the platform approval model, and Agent Gateway becomes the enforcement layer.

Layer Owns Does not own
ToolCatalogEntry Verified tool server version, discovered tool names, safety metadata, source URL. Runtime request enforcement.
ToolGrant Per-agent permission: agent X may use tools A, B, C from catalog entry Y. Networking, routing, or protocol handling.
Agent Gateway Authentication, authorization, filtered discovery, multiplexing, audit telemetry. Agent prompts, model choice, or business approval workflow.
kagent Agent CRs, model config, prompt, selected RemoteMCPServer, client-side toolNames. The final security boundary for MCP tool access.

The Policy Projection

The platform renderer takes a friendly grant and turns it into gateway-enforced CEL. The same rule affects both discovery and execution.

Human-readable grant

apiVersion: platform.kagent.dev/v1alpha1
kind: ToolGrant
metadata:
  name: cert-manager-agent-kube-readonly
  namespace: team-a
spec:
  agentRef:
    name: cert-manager-agent
  toolCatalogRef: kube-readonly-tools@v1.0.0
  allowedToolNames:
    - k8s_get_resources
    - k8s_describe_resource
    - k8s_get_events

Gateway enforcement rule

backend:
  mcp:
    authorization:
      policy:
        matchExpressions:
          - >-
            request.headers["x-kagent-namespace"] == "team-a" &&
            request.headers["x-kagent-agent"] == "cert-manager-agent" &&
            mcp.tool.target == "team-a-kube" &&
            mcp.tool.name in [
              "k8s_get_resources",
              "k8s_describe_resource",
              "k8s_get_events"
            ]
!

The headers in this demo are deliberately simple so the pattern is easy to see. For a production boundary, use signed identity, mTLS, or network policy so tenants cannot bypass kagent and spoof x-kagent-agent.

Why This Is Better Than Direct MCP Access

Direct kagent → MCP server

  • Every agent needs direct network reachability to every granted tool server.
  • toolNames narrows what the agent asks for, but it is not the central enforcement point.
  • Discovery, auth, and audit are spread across tool servers and agent config.

kagent → Agent Gateway → MCP server

  • Agents use one controlled MCP entry point per tool surface.
  • Discovery is filtered before the agent sees tools it is not allowed to use.
  • Tool execution is blocked centrally using the same policy and same telemetry path.

What We Need To Build

The upstream projects already have the important primitives. The platform work is the projection layer that keeps source-of-truth grants and runtime gateway policy in sync.

Component Responsibility Output
Onboarding workflow Quarantine a BYO MCP server, run tools/list, validate safety metadata. ToolCatalogEntry.status.verifiedTools[]
