kagent-public observability pattern

Chaos fault to Grafana MCP evidence

This page joins the implemented LitmusChaos drill path with the implemented Grafana MCP alert-enrichment path, then shows the missing approval boundary for dynamic dashboard writes.

Yes Chaos drills can inject bounded faults and route ChaosResult events to kagent.
Yes Alerts can call observability-agent and ask it to use Grafana MCP evidence.
Partial Full agent-mediated A2A depends on the model backend being healthy.
Guarded Grafana dashboard writes need a separate approved workflow and service account.

Operator flow

  1. Inject one approved fault against chaos-demo / chaos-target.
  2. Let Litmus publish the ChaosResult and Argo start the workflow.
  3. Have kagent inspect live state and classify the action tier.
  4. When an alert fires, use Grafana MCP for metrics, logs, dashboards, and deeplinks.
  5. Return an SRE-ready evidence pack plus this human-readable visual handoff.

Incident Timeline

Use this as the evidence shape for a drill, a real alert, or a combined chaos-and-alert demo.

1. Fault injected

Run one approved Litmus experiment: pod-delete or pod-cpu-hog against chaos-demo/chaos-target.

Evidence: ChaosEngine, ChaosResult, target deployment state.

2. Event captured

Argo Events watches completed ChaosResult objects and starts kagent-triage-litmus-*.

Evidence: EventSource, Sensor, Workflow phase.

3. Agent investigates

chaos-triage-agent checks the target workload and writes bounded remediation or observe-only evidence.

Evidence: agent output, deployment annotations, hive-mind ConfigMap.

4. Alert enriched

Alertmanager or Grafana Alerting routes to Argo; observability-agent queries Grafana MCP.

Evidence: datasource UID, PromQL, LogQL, dashboard panel queries.

5. SRE handoff

Return the verdict, blast radius, uncertainty, next actions, Grafana deeplinks, and local visual dashboard.

Evidence: SRE issue, generated HTML, optional approved Grafana dashboard UID.

Queries To Run

The agent should adapt these to the alert labels and always report the exact query it used.

# Prometheus or Mimir
sum by (pod) (
  increase(kube_pod_container_status_restarts_total{
    namespace="{{NAMESPACE}}"
  }[30m])
)

histogram_quantile(
  0.95,
  sum by (le, route) (
    rate(agentgateway_request_duration_seconds_bucket[5m])
  )
)
# Loki
{namespace="{{NAMESPACE}}"}
  |~ "error|fail|timeout|panic|denied|unavailable"

{namespace="kagent"}
  |~ "tool|mcp|error|timeout|model|a2a"

{namespace="argo-events"}
  |~ "sensor|eventsource|Failed|Error"

Decision Boundary

This keeps analysis fast without giving the default agent unsafe write access.

Observe only

Expected chaos result

The experiment is bounded, the workload recovered, and metrics/logs confirm no wider impact.

Human review

Ambiguous or unsafe

The namespace is not approved, the blast radius is unclear, Grafana data is incomplete, or the model route is unhealthy.

Write approval

Dashboard or remediation writes

Use a separate workflow, scoped service account, review step, and durable issue before changing Grafana or Kubernetes resources.
