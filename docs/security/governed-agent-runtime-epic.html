Governance epic

Governed agent runtime behind agentgateway

A compliance-facing control model for kagent agents that call LLMs, MCP tools, and peer agents only through governed gateway paths.

Outcome

Every agent interaction has an owner, identity, route, grant, data handling decision, audit trail, and exception path.

Boundary

Agentgateway becomes the control plane for LLM, MCP, and A2A traffic. Kubernetes policy locks down the agent and tool runtime underneath it.

Handoff

Engineering owns build and validation. Governance owns policy decisions, risk acceptance, control mapping, and sign-off evidence.

Runtime architecture

One gateway for models, tools, and agents

Agents can request access, but the platform decides. Model calls, MCP discovery, MCP execution, and A2A routes pass through policy layers that can be audited and changed through Git.

Control stack

Security is layered, not delegated to the LLM

Change gate

PR, schema validation, security review, Argo workflow, Flux reconcile, Kyverno admission.

Gateway gate

JWT, API key, OAuth or ingress identity, CEL authorization, rate limits, guardrails, telemetry.

Tool gate

Quarantine, discovered tools, catalog status, ToolGrant, explicit toolNames, denied calls.

Runtime gate

PSS restricted, no unnecessary SA token, narrow RBAC, default-deny egress, resource limits.

Evidence gate

Logs, traces, policy reports, spend dashboards, approval records, exception expiry.

Policy question Engineering control Governance evidence
Can this agent call this model? ModelConfig through agentgateway, direct provider egress denied unless exception exists. Model inventory, route policy, data residency decision, token budget.
Can this agent call this tool? Verified ToolCatalogEntry plus ToolGrant plus agentgateway MCP authorization. Grant list, gateway policy, denied-call log, catalog status.
Can this agent call another agent? A2A route policy at gateway or ingress with trusted identity and default deny. Caller/callee register, approval, route tests, audit trail.
Sandbox model

Sandbox where risk concentrates

Most agent risk is permission misuse, not container escape. The baseline is RBAC, ToolGrant, NetworkPolicy, Kyverno, and audit. Kata is reserved for higher-risk MCP and custom-code surfaces.

Baseline

Trusted read-only agents: restricted pod security, narrow tools, default-deny egress, no direct LLM provider route.

Review

Write-capable agents and broad platform tools: human approval, narrower RBAC, explicit destructive action controls.

Isolate

BYO MCP, custom-code agents, shell/code-exec tools, and quarantine servers: consider AKS Kata pod sandboxing.

Compliance backlog

Top-level epic workstreams

1

Inventory and ownership

Register agents, tools, models, grants, A2A routes, owners, data classes, expiry, and approval status.

2

Model governance

Approve providers, regions, token budgets, retention terms, direct-provider exceptions, and gateway policies.

3

MCP authorization

Quarantine tool servers, verify discovered tools, issue ToolGrants, and enforce gateway discovery filtering.

4

A2A governance

Define caller and callee trust, route approval, identity requirements, and default-deny tests.

5

Sandbox baseline

Set pod hardening, egress, service account token, RBAC, quota, and Kata tiering requirements.

6

Evidence pack

Produce diagrams, control matrix, validation output, policy reports, telemetry samples, and residual risks.

First proof

What sign-off should see first

Proof Pass condition Evidence owner
Gateway-only LLM access Direct provider call fails from agent namespace; agentgateway route succeeds. Platform engineering
Granted MCP only Agent sees only granted tools; denied tool call is blocked and logged. Platform engineering
A2A default deny Tenant agent cannot reach privileged platform agent without approved route policy. Platform security
Egress isolation Agent and tool workloads cannot reach IMDS, direct model providers, arbitrary external HTTP, or cross-namespace services unless allowlisted. This is workload isolation, not the gateway data-plane egress policy. Platform security
Governed change path PR, workflow, Flux state, Kyverno report, and telemetry all correlate to one change. Governance team
