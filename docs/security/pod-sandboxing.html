Security Decision

Pod Sandboxing for kagent

Use AKS Kata isolation for untrusted or write-capable execution paths. Keep trusted read-only agents focused on RBAC, tool grants, NetworkPolicy, and audit controls.

Current AKS names

--workload-runtime KataVmIsolation

runtimeClassName: kata-vm-isolation

These replace stale kata-mshv-vm-isolation examples from older material.

Decision Matrix

No

Controller and UI

Trusted platform components with no tenant-supplied execution path. Harden them normally and keep permissions tight.

Maybe

kagent-tool-server

Worth piloting when its Kubernetes or Azure permissions cannot be tightly bounded, or when prompt-driven tools can execute powerful commands.

Yes

BYO MCP and Code Tools

Use Kata for quarantine servers, tenant-supplied images, custom-code agents, and future Python REPL or code-interpreter tools.

What Kata Changes

Threat Normal Hardened Pod Kata Pod
Prompt injection using allowed tools Still dangerous Still dangerous
Kernel or runtime escape Shared host kernel is exposed Workload runs behind a guest VM kernel
Privileged container Should be blocked by policy Root is contained inside the guest VM if allowed
Tenant-supplied code Higher blast radius if escaped Better host and neighbor isolation

Minimum Rollout

Start with quarantine or tool-server workloads. Measure cold start, memory, and request latency before widening.

az aks nodepool add \ --cluster-name <aks> \ --resource-group <rg> \ --name katapool \ --os-sku AzureLinux \ --workload-runtime KataVmIsolation \ --node-vm-size Standard_D4s_v5
spec: template: spec: runtimeClassName: kata-vm-isolation

Guardrails Still Matter More

  • Separate read-only agents from write-capable agents.
  • Use Kyverno or OPA to restrict tool grants and dangerous verbs.
  • Lock egress with NetworkPolicy and route LLM traffic through agentgateway.
  • Avoid hostPath on sandboxed pods.
  • Set CPU and memory requests/limits deliberately for Kata overhead.
